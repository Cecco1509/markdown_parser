## 1. The LR item data structure

An LR(1) item is conceptually `(production, dot position, lookahead set)`. The key design decision is **how to split the "core" from the lookahead**, because the whole LALR construction hinges on that split:

- **closure/goto** and canonical-collection dedup care about the *full* item (core + lookahead).
- the **LALR merge** groups states by *core only* (production + dot), ignoring lookahead, then unions lookaheads.

So I want a representation where "the core" is a cheap, comparable, hashable value, and the lookahead hangs off it. My choice:

```cpp
// The LR(0) CORE of an item: which production, and how far the dot has moved.
struct Item {
  int prod;   // index into the production vector
  int dot;    // 0 .. rhs.size()
  auto operator<=>(const Item&) const = default;   // ordered => usable as a map key
};

// Lookahead terminals. Typedef'd so the backing store is swappable.
using Lookahead = std::set<int>;   // terminal ids

// An LR(1) item set = each core mapped to its lookahead set.
// Merging two cores' lookaheads is just a set-union on the value.
using ItemSet = std::map<Item, Lookahead>;

struct State {
  ItemSet items;
  std::map<int, int> trans;   // symbol id -> target state id
};
```

Why these specific choices:

- **`Item` = just `{prod, dot}` as two ints.** The production's symbols are *not* copied into the item — they're looked up via `prod` in the grammar. This keeps items tiny and makes core-equality a trivial 2-int compare, which the merge does constantly. `operator<=>` default lets `Item` be a `std::map`/`std::set` key with no boilerplate.
- **Lookahead separated as the map *value*, not baked into `Item`.** This is the crux: `ItemSet = map<core, lookahead>` means (a) LR(1) closure just unions into the value, (b) the LALR merge compares *keys only* and unions *values*, and (c) an item set prints/deduplicates cleanly. If I'd folded the lookahead into the item key, every lookahead change would churn the key and the merge would be painful.
- **`Lookahead = std::set<int>` behind a typedef.** Our grammar has ~22 terminals, so performance is irrelevant and correctness/readability win — set-union is `std::set_union`, comparison is built-in. The typedef means if it ever mattered I could swap it for a dynamic bitset (`vector<uint64_t>`, word-wise OR) without touching any call site. I'm not doing that now; it'd be premature.
- **Canonical-collection dedup = `std::map<ItemSet, int>`** (item set → state id). Because `ItemSet` is an ordered `std::map`, it's directly comparable, so I get dedup with no hand-written hashing. For the **merge**, I project each state onto its key set (`std::set<Item>`) and group states that share it.

That's the whole item model: **thin integer core + separated lookahead value**, chosen specifically so closure, dedup, and the LALR merge each stay simple.

## The real tradeoff

| | Canonical LR(1) | LALR(1) |
|---|---|---|
| Power | Strictly larger grammar class | Subset — every LALR(1) grammar is LR(1), not vice versa |
| States | Many more (the classic blowup) | Merged down to the LR(0)/SLR count |
| Conflicts | None beyond genuine grammar ambiguity | The merge can introduce **reduce/reduce conflicts** that LR(1) doesn't have |
| Why it exists | The "correct" bottom-up method | A **1970s memory optimization** — shrink the table when RAM was scarce |

The one and only thing LALR buys over canonical LR(1) is **fewer states → smaller tables.** That was decisive in 1971. For us: our grammar is ~30 productions and ~22 terminals. Canonical LR(1) might be a few hundred states vs ~100 for LALR — a few kilobytes of `constexpr` tables generated at build time, with **zero runtime difference**. It's a non-issue.

## The part that actually changes my mind

My plan was **"build the canonical LR(1) automaton, *then* merge to LALR."** So I'm constructing the full LR(1) machine either way. Given that:

- **Stopping before the merge gives me canonical LR(1) for *less* code** — I drop the merge/lookahead-union step entirely.
- The merge is the **subtlest, most bug-prone phase** (unioning lookahead sets across fused cores), and it's the *only* place a grammar that's cleanly LR(1) can suddenly sprout a **"mysterious" reduce/reduce conflict** — two LR(1) states with the same core but different lookaheads get fused, and their reduces collide on a shared lookahead that neither original state had a problem with.

So for our situation, LALR is *more* implementation work that yields a *less* powerful parser and *adds* a class of failure — in exchange for saving memory we don't need. That's a bad trade here. The historical reason to prefer LALR is simply absent at this scale.

## My recommendation

I'd switch to **canonical LR(1)**: since we're building that automaton anyway, we skip the merge, get a strictly more powerful and simpler generator, and eliminate merge-induced conflicts. The only reasons to *keep* LALR would be pedagogical — if a goal of the project is specifically to demonstrate the LALR merge or to mirror Bison's exact algorithm.

> Explain how you want to build the ItemSet

The `ItemSet` is built by two operations — **`closure`** (completes a set) and **`goto`** (transitions to the next set) — driven by a worklist that discovers all states. `closure` is the heart of it, because that's where LR(1) lookaheads are computed and propagated. Here's exactly how I'll build it against our `ItemSet = std::map<Item, std::set<int>>`.

## Precomputed helpers (done once, before any item set)

- `nullable[symbol]` and `FIRST[symbol]` (terminals: `FIRST(t) = {t}`; nonterminals: standard fixpoint).
- `firstOfSeq(rhs, from)` → `(std::set<int> terminals, bool nullable)`: walk `rhs[from..]`, union each symbol's `FIRST` minus ε; stop at the first non-nullable symbol; report `nullable = true` iff *all* of them (or none) are nullable. This is what turns "the symbols after the dot" into lookaheads.

## `closure(ItemSet& I)` — in-place fixpoint

For an item `[A → α · B β, la]` where `B` is a nonterminal, LR(1) says: add `[B → · γ, b]` for every production `B → γ` and every `b ∈ FIRST(β · la)`. Concretely:

```
closure(I):                      # I starts as the kernel, gets completed in place
  worklist ← every core in I
  while worklist not empty:
    core ← pop;  la ← I[core]                 # read CURRENT lookahead
    P ← productions[core.prod]
    if core.dot ≥ |P.rhs|: continue           # dot at end, nothing after it
    B ← P.rhs[core.dot]
    if B is a terminal: continue              # only nonterminals expand

    (firstBeta, betaNullable) ← firstOfSeq(P.rhs, core.dot + 1)   # β = symbols after B
    lookForB ← firstBeta
    if betaNullable: lookForB ← lookForB ∪ la                     # FIRST(β·la), β nullable ⇒ inherit la

    for each production p with lhs == B:
      child ← Item{ p.index, 0 }               # [B → · γ]
      grew ← union_into(I[child], lookForB)     # merge lookaheads by CORE
      if grew or child was newly inserted:
        push child onto worklist                # its lookahead changed ⇒ re-propagate
```

Two subtleties this structure handles cleanly, and they're the whole reason `Item{prod,dot}` is the map **key** with the lookahead as the **value**:

1. **Merge-by-core.** When two different items both introduce `[B → · γ]` with different lookaheads, `union_into(I[child], …)` fuses them into one entry with the unioned lookahead set — never two competing copies.
2. **Re-propagation on growth.** A child's lookahead can grow *after* it was first added (because some other item feeds it more terminals). `union_into` returns whether the set actually grew; if so we re-queue that core so its own children get the newly-arrived lookaheads. That's the fixpoint — it terminates because lookahead sets only ever grow and are bounded by the terminal count.

`union_into(dst, src)` is just: insert each element, track if `dst.size()` increased.

## `goto(const ItemSet& I, int X) → ItemSet`

Advance the dot over symbol `X`, then close:

```
goto(I, X):
  J ← {}
  for (core, la) in I:
    P ← productions[core.prod]
    if core.dot < |P.rhs| and P.rhs[core.dot] == X:
      union_into( J[ Item{core.prod, core.dot+1} ], la )   # carry lookaheads forward
  closure(J)
  return J
```

## Canonical collection — discovering every `ItemSet`

Worklist over states, with dedup by the *whole closed* `ItemSet` (keys + lookaheads), since in LR(1) two states are identical only if their items **and** lookaheads match:

```
K0 ← { Item{augmented S'→·document, 0} : {END} }   # $ is the only start lookahead
states ← [ State{ closure(K0) } ]
index  ← { states[0].items → 0 }                    # ItemSet -> state id (map is comparable)
work   ← [0]
while work:
  s ← pop;  I ← states[s].items
  for each X in { P.rhs[core.dot] : (core,_)∈I, core.dot<|P.rhs| }:   # symbols right of a dot
    J ← goto(I, X)
    if J empty: continue
    t ← index[J]  if present  else  (append State{J}; index[J]=new id; enqueue)
    states[s].trans[X] = t
```

Because `ItemSet` is an ordered `std::map`, `index` is just a `std::map<ItemSet,int>` — no hand-rolled hashing, and dedup is a plain lookup.

## Tiny worked slice (state 0)

Kernel `[S' → · document, $]`. Closure sees `document` after the dot → adds `document : · header stmts` with lookahead `$` (β empty ⇒ inherit `$`). That has `header` after the dot, β = `stmts`; `firstOfSeq(stmts)` includes `NEWLINE` (and, since `stmts` is nullable via `%empty`, also inherits `$`). So we add `header : · GRAPH NEWLINE` and `header : · GRAPH DIR NEWLINE` with lookaheads `{NEWLINE, $}`. Now `GRAPH` (a terminal) sits after the dot in two items → `goto(state0, GRAPH)` becomes the kernel of the next state. That's the machine building itself.

That's the full ItemSet construction: **`closure` = a lookahead fixpoint over a core→lookahead map**, **`goto` = advance-then-close**, **collection = worklist + map-dedup**. The item's thin `{prod,dot}` core as the map key is what makes merge-by-core and growth-detection fall out for free.
